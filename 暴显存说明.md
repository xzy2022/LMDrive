按照环境说明.md运行测评代码暴显存。

根因很直接：你现在这套 LMDrive 配置本身就不是 8GB 显卡能稳跑的，OOM 不是偶发，而是模型体量超了。

**为什么会爆显存**

你的 agent 配置里明确用了 `llava-v1.5-7b`，见 [lmdriver_config_docker.py](/workspace/LMDrive/leaderboard/team_code/lmdriver_config_docker.py#L23)。  
在模型实现里，它会先加载整套 LLM，再加载视觉编码器和 Q-Former，最后在 [lmdriver_agent.py](/workspace/LMDrive/leaderboard/team_code/lmdriver_agent.py#L220) 直接执行整网 `self.net.cuda()`，也就是把整套模型一次性搬到单张 GPU 上。

我本地查到的几个关键事实是：

- 你的 GPU 实际总显存是 `8151 MiB`，也就是日志里说的 `7.50 GiB`
- 运行当下显存状态是：`7562 MiB used / 120 MiB free`
- 当前 `python3` 进程已经占了 `6236 MiB`
- 你加载的 `llava-v1.5-7b` 权重目录本身就有 `13G`
- 视觉编码器目录还有 `360M`
- 运行时还会加载一个 `Q-Former`，日志里也显示了 `model.safetensors 440M`

这就意味着：

- 单是 `7B` 这套 LLM 的权重体量，量级就已经在 `13GB` 左右
- 你当前代码虽然把 LLM 用 `float16` 加载了，但没有做 `8bit/4bit` 量化，也没有 `device_map`/CPU offload
- `freeze`、`eval()`、`no_grad()` 只能省梯度和部分激活，省不了模型参数本身占的显存
- 所以在 `self.net.cuda()` 过程中，模型还没搬完就已经把 8G 卡塞满了，最后多申请 `86 MiB` 就直接 OOM

换句话说，不是“8G 怎么连 86MB 都申请不到”，而是“前面已经快塞满了，这 86MB 是压死骆驼的最后一根稻草”。

**为什么你觉得只有 6GB 多，但还是说 13GB 不够**

这两个数字不矛盾：

- `nvidia-smi` 里看到的 `6236 MiB`，是 OOM 时已经成功搬上 GPU 的那一部分
- 还没搬过去的剩余层，在继续 `.cuda()` 时申请失败了
- 所以它不是“完整模型只需要 6.2GB”，而是“搬到 6.2GB 左右时就已经快顶满 8G 了，后续再搬就爆了”

另外，即使先不算 CARLA，这个模型也已经不适合 8G。  
如果 CARLA 和 agent 用的是同一张卡，那只会更紧，不会更松。

**结论**

这套配置对 8GB 卡来说，瓶颈不是碎片，而是模型本体太大。`PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` 这种设置最多缓解碎片，救不了一个本来就装不下的 `7B + 视觉编码器 + Q-Former` 组合。

想跑起来，通常只剩这几条路：

- 换更大显存的卡，至少 `16GB`，更稳妥是 `24GB`
- 把 LLM 改成更小的模型
- 给当前代码加 `8bit/4bit` 量化，或者做 CPU offload
- 把 CARLA 和推理拆到不同 GPU
